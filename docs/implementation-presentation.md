# 구현 설명과 SQL 처리기 특징

이 문서는 코드리뷰가 아니라 구현 설명을 빠르게 진행하기 위한 발표용 정리 문서다.
한 화면에서 전체 구조, 핵심 흐름, SQL 처리기의 특징과 차별점, 장단점을 함께 설명할 수 있도록 구성했다.

## 1. 한 줄 소개

이 프로젝트는 `books` 단일 테이블만 다루는 C11 기반 미니 SQL 처리기다.
입력 SQL을 직접 파싱해서 실행하고, `id` 조건 조회는 B+ 트리로 최적화하며, 데이터는 바이너리 파일로 저장한다.

## 2. 먼저 설명할 핵심 메시지

- 이 프로젝트는 범용 DBMS가 아니라 읽기 쉬운 SQL 처리기 데모다.
- 테이블은 `books` 하나로 고정되어 있고 스키마도 `id`, `title`, `author`, `genre`로 단순화했다.
- `SELECT`, `INSERT`, 배치 실행, 롤백, 바이너리 파일 I/O를 직접 구현했다.
- `id` 기반 조회는 B+ 트리 인덱스를 사용하고, 그 외 조회는 선형 탐색을 사용한다.
- 데이터 파일에는 row만 저장하고, B+ 트리는 실행 시 메모리에 다시 구성한다.

## 3. 전체 동작 흐름

설명 순서는 아래 흐름으로 잡으면 가장 이해가 빠르다.

1. `main.c`
2. `cli.c` / `batch.c`
3. `lex.c`
4. `parse.c`
5. `exec.c`
6. `store.c`
7. `bpt.c`
8. `tests/`

전체 흐름은 다음과 같다.

1. 프로그램이 시작되면 CLI 옵션을 읽는다.
2. 데이터 파일 경로를 정하고 `db_load()`로 메모리 DB를 준비한다.
3. 배치 SQL을 문자열 또는 파일에서 읽는다.
4. `split_sql()`이 세미콜론 기준으로 여러 문장으로 나눈다.
5. 각 문장은 lexer에서 토큰으로 바뀌고 parser에서 `Qry` 구조체로 바뀐다.
6. 실행기는 `SELECT` 또는 `INSERT`를 수행한다.
7. `SELECT`는 조건에 따라 B+ 트리 또는 선형 탐색으로 결과를 찾는다.
8. `INSERT`가 포함된 배치는 모두 성공했을 때만 저장한다.
9. 중간에 실패하면 메모리 상태와 `next_id`, B+ 트리를 롤백한다.

## 4. 파일별 역할 설명

### `src/main.c`

- 프로그램의 진입점이다.
- 옵션 파싱, 데이터 로드, SQL 입력 확보, 배치 실행, 결과 출력까지 전체 흐름을 연결한다.
- 발표할 때는 "오케스트라 지휘자 역할"이라고 설명하면 이해가 쉽다.

### `src/cli.c`

- `--mode`, `--batch`, `--file`, `--data`, `--summary-only` 옵션을 해석한다.
- 대화형 입력 모드와 파일 입력 모드를 나누는 역할을 한다.

### `src/batch.c`

- SQL 파일을 읽는다.
- 일반 SQL 텍스트와 `QSQL` 바이너리 포맷을 둘 다 지원한다.
- 여러 문장을 세미콜론 기준으로 나누되, 문자열 안의 세미콜론은 구분자로 처리하지 않는다.

### `src/lex.c`

- SQL 문자열을 토큰 목록으로 바꾼다.
- 키워드, 식별자, 숫자, 문자열, 기호를 구분한다.
- 즉 SQL 문장을 컴퓨터가 이해할 수 있는 작은 조각으로 나누는 단계다.

### `src/parse.c`

- lexer가 만든 토큰 목록을 `Qry` 구조체로 변환한다.
- 이 프로젝트에서 지원하는 제한된 SQL 문법만 받아들인다.
- `SELECT`, `INSERT`, `WHERE column = value`, `WHERE id BETWEEN a AND b`를 처리한다.

### `src/exec.c`

- 실제 SQL 실행을 담당하는 핵심 모듈이다.
- 파싱된 질의를 받아 조회 또는 삽입을 수행한다.
- 배치 출력 버퍼링, 성공 시 저장, 실패 시 롤백까지 책임진다.

### `src/store.c`

- 메모리 DB와 바이너리 파일 저장소를 연결한다.
- row 추가, 데이터 저장, 데이터 로드, 인덱스 재구성을 담당한다.
- 파일에는 `BKDB` 포맷으로 row와 `next_id`를 저장한다.

### `src/bpt.c`

- `id` 전용 B+ 트리 인덱스를 구현한다.
- 단건 조회와 범위 조회를 지원한다.
- 저장소가 아니라 검색 가속을 위한 메모리 인덱스라는 점이 핵심이다.

## 5. SQL 처리기의 주요 특징

### 5.1 단일 테이블 고정 구조

- `books` 테이블 하나만 지원한다.
- 범용성을 줄인 대신 코드와 설명이 단순하다.
- 교육용, 데모용으로 구조를 파악하기 좋다.

### 5.2 제한된 문법만 직접 파싱

- 복잡한 SQL 문법 전체를 구현하지 않는다.
- 필요한 문법만 lexer와 parser로 직접 처리한다.
- 덕분에 파싱 흐름을 처음부터 끝까지 추적하기 쉽다.

### 5.3 `id` 조건은 B+ 트리, 나머지는 선형 탐색

- `WHERE id = n`은 B+ 트리 단건 조회를 사용한다.
- 현재 구현 기준으로 `WHERE id BETWEEN a AND b`도 B+ 트리 범위 조회를 사용한다.
- `title`, `author`, `genre`, 전체 조회는 선형 탐색을 사용한다.
- 어떤 경우에 인덱스를 쓰고 어떤 경우에 스캔하는지가 명확하다.

### 5.4 배치 단위 실행과 출력 버퍼링

- 여러 SQL 문장을 한 번에 실행할 수 있다.
- 출력은 중간에 바로 내보내지 않고 버퍼에 모아 둔다.
- 배치 전체가 성공했을 때만 최종 출력한다.

### 5.5 실패 시 롤백

- 배치 시작 전 `len`과 `next_id`를 저장해 둔다.
- 실패하면 원래 길이와 `next_id`로 복원한다.
- 이후 B+ 트리를 다시 구성해서 메모리 상태를 맞춘다.

### 5.6 바이너리 파일 저장

- 데이터를 `BKDB` 바이너리 포맷으로 저장한다.
- 고정 길이 문자열을 사용해서 I/O가 단순하다.
- 저장은 `.tmp` 파일을 먼저 쓰고 교체하는 방식이라 실패에 비교적 강하다.

### 5.7 메모리 인덱스 재구성 방식

- B+ 트리 자체는 파일에 저장하지 않는다.
- 프로그램 시작 시 row를 읽고 `db_reidx()`로 다시 만든다.
- 따라서 인덱스는 영속 데이터가 아니라 실행 중 보조 구조다.

## 6. 이 SQL 처리기의 차별점

### 6.1 교육용으로 흐름이 끝까지 드러난다

- SQL 입력부터 파일 저장까지 전 과정을 한 저장소 안에서 모두 볼 수 있다.
- 실제 DBMS에서 숨겨지는 부분을 직접 구현해 이해하기 쉽다.

### 6.2 인덱스 사용 범위가 명확하다

- 모든 조회를 무조건 빠르게 만들려는 구조가 아니다.
- `id` 조건만 인덱스로 최적화하고, 나머지는 의도적으로 선형 탐색으로 남겼다.
- 그래서 인덱스의 효과와 선형 탐색의 차이를 설명하기 좋다.

### 6.3 롤백 방식이 단순하고 설명 가능하다

- undo log나 MVCC 같은 복잡한 기법을 쓰지 않는다.
- 현재 변경 연산이 `INSERT` 중심이라서 row 길이 복구와 인덱스 재구성만으로 복구할 수 있다.
- 구현이 짧고 발표에서 설명하기 쉽다.

### 6.4 저장 포맷이 명시적이다

- `QSQL`, `BKDB`라는 고정된 바이너리 포맷을 문서와 코드에서 바로 확인할 수 있다.
- 텍스트 입력과 바이너리 입력, 메모리 구조와 디스크 구조를 비교하기 좋다.

## 7. 장점

- 구조가 작고 읽기 쉬워서 전체 흐름을 설명하기 좋다.
- SQL 처리기의 핵심 요소인 lexer, parser, executor, storage, index를 모두 직접 보여줄 수 있다.
- `id` 인덱스와 선형 탐색의 차이를 실험적으로 보여주기 좋다.
- 바이너리 포맷과 파일 교체 저장 방식 덕분에 저장 흐름이 명확하다.
- 배치 버퍼링과 롤백으로 "실패 시 상태 복구" 개념을 설명하기 좋다.
- 테스트 코드가 있어서 기능 설명 뒤에 근거를 바로 제시할 수 있다.

## 8. 단점과 트레이드오프

- 단일 테이블 구조라 범용성이 낮다.
- SQL 문법 지원 범위가 작아서 일반적인 SQL 엔진처럼 쓰기는 어렵다.
- `author`, `genre`, `title` 조회는 여전히 선형 탐색이라 데이터가 커지면 느려진다.
- B+ 트리를 파일에 저장하지 않기 때문에 실행할 때마다 다시 구성해야 한다.
- 롤백 방식이 단순한 대신 `UPDATE`, `DELETE`, 동시성 같은 확장에는 바로 맞지 않는다.
- 고정 길이 문자열은 구현은 단순하지만 디스크 공간 효율은 떨어질 수 있다.

## 9. 발표에서 강조하면 좋은 설계 선택

- 범용성을 줄이고 설명 가능성을 높였다.
- 인덱스를 모든 컬럼에 붙이지 않고 `id`에만 적용해 요구사항과 학습 포인트를 맞췄다.
- 데이터 파일은 단순하게 유지하고, 인덱스는 실행 시 재구성해서 구현 복잡도를 낮췄다.
- 저장은 임시 파일 교체 방식으로 처리해 부분 저장 위험을 줄였다.
- 배치 실행은 성공 또는 실패가 분명하게 보이도록 버퍼링과 롤백을 사용했다.

## 10. 발표용 마무리 문장

이 SQL 처리기의 핵심은 "작지만 끝까지 보이는 구조"다.
복잡한 DBMS의 모든 기능을 구현하는 대신, SQL 입력이 어떻게 해석되고 실행되며 저장되는지, 그리고 인덱스와 롤백이 어떤 역할을 하는지를 직접 따라갈 수 있게 만든 것이 가장 큰 특징이다.
